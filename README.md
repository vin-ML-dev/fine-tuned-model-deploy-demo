# TechNova Company Chatbot — Production ML Pipeline

Fine-tune a small LLM on a custom company knowledge base (QLoRA on a Colab T4 GPU), then productionize it step by step: GGUF conversion → vLLM/FastAPI serving → local Kubernetes → CI/CD → monitoring.

**This repo currently covers Phase 1 (Data) and Phase 2 (Fine-Tuning).** Later phases will be added on top of it.

---

## Tech decisions (and why)

| Decision | Choice | Reason |
|---|---|---|
| Base model | `Qwen/Qwen2.5-1.5B-Instruct` | Fits easily on a 16 GB T4 with QLoRA, non-gated (no HF access request needed), strong instruction-following for its size |
| Fine-tuning | QLoRA (4-bit NF4) + PEFT LoRA | Trains only ~1% of parameters → fast + low VRAM |
| Precision | fp16 (NOT bf16) | The T4 GPU has no bf16 support |
| Tracking | TensorBoard | Works out-of-the-box in Colab, no account needed |
| Later | GGUF, vLLM, FastAPI, k8s, CI/CD | Phases 3–5 |

## Phase 1 — Problem & Data

### Step 1: Task definition
- **Task:** Company Q&A chatbot ("NovaBot") for TechNova Solutions — answers questions about the company, products, policies (refund, privacy, leave, travel, security, etc.).
- **Success metric:** low test loss/perplexity + factually correct answers on held-out questions.
- **Serving targets (for later phases):** p95 latency < 2s for a 200-token answer on a single GPU.

### Step 2: The knowledge base
`data/raw/company_knowledge_base.txt` is the single source of truth. It contains all 14 requested sections: company overview, mission/vision/values, products & services, business hours, customer support, refund policy, privacy policy, data retention, employee handbook, leave policy, code of conduct, information security, travel policy, and performance reviews.

### Step 3: Dataset generation
```bash
python scripts/generate_dataset.py
```
What it does:
- Contains ~90 curated question groups (each with 2–3 human-written paraphrases) whose answers come **only** from the knowledge base — this prevents the model from learning hallucinated facts.
- Augments each question with natural prefixes ("Hi, ...", "Quick question — ...") so the model generalizes across phrasings.
- Includes **out-of-scope refusal examples** ("What's the weather?") so the bot learns to stay on-topic — a real production requirement.
- Writes everything in the standard chat `messages` format (system / user / assistant), which `trl`'s SFTTrainer consumes directly using the model's own chat template.

Output: `data/processed/dataset_full.jsonl` (~380 examples).

### Step 4: Preprocessing & splitting
```bash
python scripts/preprocess.py
```
What it does:
- Normalizes whitespace in all messages.
- **Deduplicates** on a normalized version of the user question (lowercased, punctuation stripped) — duplicate training questions inflate metrics and waste compute.
- **Stratified split** per policy section into train (85%) / val (10%) / test (5%) — stratification guarantees every policy topic appears in every split, so eval loss reflects all topics.
- Writes `dataset_stats.json` — a lightweight data-versioning record (counts per section, split sizes, average answer length). Commit this file with each dataset change; for larger projects, swap in DVC.

## Phase 2 — Fine-Tuning

### Step 5: Setup on Google Colab (T4)

1. Runtime → Change runtime type → **T4 GPU**
2. Clone / upload the repo, then:
```bash
pip install -q -r requirements.txt
```
> Do **not** reinstall torch on Colab — it ships with the correct CUDA build.

### Step 6: Configuration
Everything lives in `configs/training_config.yaml`. Key T4-specific choices:
- `fp16: true` and `bnb_4bit_compute_dtype: float16` — **T4 does not support bf16**; using bf16 will crash or silently degrade.
- `load_in_4bit + nf4 + double_quant` — the QLoRA recipe; the 1.5B model uses ~1.5 GB VRAM quantized.
- `gradient_checkpointing: true` + `paged_adamw_8bit` — trade compute for memory; keeps peak usage well under 16 GB.
- `per_device_train_batch_size: 4` × `gradient_accumulation_steps: 4` = effective batch 16.
- `report_to: tensorboard` — experiment tracking (step 6 of the roadmap).

### Step 7: Train
```bash
python scripts/train_qlora.py --config configs/training_config.yaml
```
What happens inside:
1. Loads the base model in 4-bit and calls `prepare_model_for_kbit_training` (enables gradient flow through the quantized weights).
2. Attaches LoRA adapters (`r=16`, all attention + MLP projections) — prints trainable params (~1%).
3. Trains with **checkpointing every epoch**, `save_total_limit=2`, **early stopping** (patience 2 on `eval_loss`), and `load_best_model_at_end=true`.
4. Saves the **best adapter** to `outputs/qlora-technova/final_adapter/` and writes `model_card.json` (base model, eval loss, LoRA config) — a minimal **model registry entry** (step 10). In a larger setup you'd push this to MLflow or the HF Hub instead.

Expected runtime on T4: roughly 10–20 minutes for ~320 train examples × 4 epochs.

Watch training curves:
```bash
%load_ext tensorboard
%tensorboard --logdir outputs/qlora-technova
```

### Step 8: Evaluate
```bash
python scripts/evaluate.py                  # test loss + 4 sample generations
python scripts/evaluate.py --compare_base   # also regression-test vs the raw base model
```
What it checks:
- **Quantitative:** average test-set loss and perplexity of the fine-tuned model.
- **Qualitative:** greedy-decoded answers vs the gold answers (spot-check factual accuracy — e.g., "refund within 7 days", "26 weeks maternity leave").
- **Regression vs baseline** (`--compare_base`): confirms fine-tuning actually improved over the raw base model.
- Writes `outputs/qlora-technova/eval_report.json` for record-keeping.

## Phase 3 — Optimization for Serving

Run these after training (on the pod, or locally if you downloaded the adapter).

### Step 11a: Merge the LoRA adapter
```bash
python scripts/merge_adapter.py            # bf16 (Ampere+ GPUs / RunPod 3090)
python scripts/merge_adapter.py --dtype float16   # for T4 / older GPUs
```
Loads the base model in **full precision** (never merge into 4-bit weights — it's lossy), applies the adapter, merges, saves to `outputs/merged-technova-1.5b/` (~3 GB), and runs a sanity-check generation. vLLM (Phase 4) will serve this merged model directly.

### Step 11b/12: Convert to GGUF + quantize
```bash
bash scripts/convert_to_gguf.sh
```
Clones llama.cpp, converts the merged model to `technova-1.5b-f16.gguf`, builds `llama-quantize`, and produces `technova-1.5b-Q4_K_M.gguf` (~1 GB). Q4_K_M is the standard best quality-vs-size quantization. GGUF gives you a CPU-friendly fallback deployment (e.g., running the bot on a laptop or a cheap VM without GPU).

Smoke test:
```bash
llama.cpp/build/bin/llama-cli -m outputs/gguf/technova-1.5b-Q4_K_M.gguf \
  -p "What is TechNova's refund policy?" -n 100
```

### Step 13: Benchmark (baseline SLO numbers)
```bash
python scripts/benchmark.py --runs 16
```
Measures p50/p95 end-to-end latency, tokens/sec, and peak GPU memory with plain `transformers` generation. Saves `outputs/benchmark_baseline.json` — this is your **baseline** to prove vLLM's improvement in Phase 4.

### What to download before killing the pod
- `outputs/merged-technova-1.5b/` (~3 GB) — needed for vLLM in Phase 4
- `outputs/gguf/technova-1.5b-Q4_K_M.gguf` (~1 GB) — CPU fallback
- `outputs/benchmark_baseline.json`

## Phase 4 — Serving: engine + FastAPI wrapper + Docker

Architecture (sidecar pattern — the API layer is engine-agnostic):
```
client ──> FastAPI wrapper (:8000) ──> engine (:8001)
                                        ├── GPU: vLLM  (serves HF safetensors repo)
                                        └── CPU: llama.cpp server (serves Q4_K_M GGUF)
```
Both engines expose the same OpenAI-compatible API, so one wrapper serves both.

### What the FastAPI wrapper (`serving/app/main.py`) adds
- **Request validation** — Pydantic schema; message length limits, history capped at 20 turns
- **Server-side system prompt** — the NovaBot prompt is enforced in the API, clients can't bypass it (and can't hit the raw-prompt bug)
- **Streaming** — pass `"stream": true` to get SSE token streaming
- **Timeout & upstream error handling** — 504 on engine timeout, 503 when engine down, 502 on engine errors
- **Rate limiting** — sliding-window per client IP (in-memory; moves to Redis in Phase 6)
- **`/healthz` vs `/readyz`** — liveness ("process alive") vs readiness ("engine loaded and reachable"); these map 1:1 to Kubernetes probes in Phase 5
- **Structured JSON logs** with request IDs and latency — Phase 7's log pipeline consumes these

### Run it — GPU path (needs NVIDIA GPU + nvidia-container-toolkit)
```bash
export HF_TOKEN=hf_xxx   # if your HF repos are private
docker compose -f serving/docker-compose.gpu.yml up --build
```

### Run it — CPU path (any machine, uses the GGUF)
```bash
export HF_TOKEN=hf_xxx
docker compose -f serving/docker-compose.cpu.yml up --build
```
First start downloads the model from your HF Hub repos into a named volume (cached afterward — this is the "Hub as model registry" pattern; the image contains no model weights).

### Run it — on a RunPod GPU pod (no Docker available!)
RunPod pods are containers themselves — you **cannot run a Docker daemon inside them**.
So on RunPod, run both processes directly:
```bash
export HF_TOKEN=hf_xxx
bash serving/run_on_runpod.sh          # starts vLLM (:8001) + FastAPI (:8000)
bash serving/run_on_runpod.sh stop     # stop both
```
To reach the API from your laptop: add **8000** as an HTTP port in the RunPod
console, then call `https://<POD_ID>-8000.proxy.runpod.net/v1/chat`.
The Docker compose files above are for machines with a real Docker daemon
(your laptop, a VPS, a cloud VM) — and they define the images Phase 5 deploys to K8s.

### Test
```bash
curl -s localhost:8000/readyz
curl -s -X POST localhost:8000/v1/chat -H 'Content-Type: application/json' \
  -d '{"message": "What is the refund policy for monthly subscriptions?"}'

python serving/test_api.py       # full smoke test suite
```

## Phase 5 — Kubernetes + CI/CD (local kind cluster, engine on RunPod)

Architecture:
```
user -> kubectl port-forward -> Service -> api pods (2..6, HPA) -> RunPod vLLM (GPU)
                 GitHub push -> Actions: tests -> build image -> push ghcr.io
                 laptop:      bash k8s/deploy_release.sh <sha>  (pull-based CD)
```

### Step 19: Cluster + first deploy
Install [minikube](https://minikube.sigs.k8s.io/docs/start/) and kubectl, fill `serving/.env` (RunPod endpoint + key), then one command:
```bash
bash k8s/setup_minikube.sh
```
It starts minikube, enables the metrics-server addon, builds the API image inside minikube's docker daemon, creates the secret from `.env`, and applies all manifests. Then:
```bash
kubectl -n novabot get pods -w                       # wait for 2/2 Ready
kubectl -n novabot port-forward svc/novabot-api 8080:80
curl -s localhost:8080/readyz
python serving/test_api.py --base http://localhost:8080
```

### The manifests (`k8s/`)
- `00-namespace.yaml` — isolation boundary for all NovaBot resources
- `10-secret.example.yaml` — RunPod credentials pattern (real secret created from `.env`, never committed)
- `20-deployment.yaml` — 2 replicas, rolling updates (`maxUnavailable: 0` = zero downtime), liveness→`/healthz`, readiness→`/readyz`, CPU/memory requests+limits
- `30-service.yaml` — stable ClusterIP load-balancing across ready pods
- `40-hpa.yaml` — autoscaling 2→6 replicas at 70% CPU (of *requests*), with scale-down stabilization

### Step 20: Watch the autoscaler
```bash
kubectl -n novabot get hpa -w
# in another terminal, generate load (install 'hey' or use a curl loop):
hey -z 60s -c 30 http://localhost:8080/healthz
```

### Step 21: CI/CD (`.github/workflows/ci-cd.yml`)
- Every push/PR: **unit tests** (`tests/test_app.py` — 9 tests, no engine or secrets needed: validation, rate limiting, 503 translation, system-prompt injection)
- Push to main: **build + push** the image to GitHub Container Registry, tagged `:<git-sha>` (immutable) + `:latest`
- Deploy is **pull-based** (GitHub can't reach your laptop): `GHCR_OWNER=<user> bash k8s/deploy_release.sh <sha>`
- First time: make the GHCR package public (repo → Packages → settings) or create a pull secret:
  `kubectl -n novabot create secret docker-registry ghcr-pull --docker-server=ghcr.io --docker-username=<user> --docker-password=<PAT>` and add `imagePullSecrets` to the deployment.

### Step 22: Rollout + rollback
```bash
kubectl -n novabot rollout history deploy/novabot-api
bash k8s/deploy_release.sh rollback        # one-command rollback
```

### Step 23 note: gateway & auth
`port-forward` stands in for an Ingress locally. Client-facing API-key auth is added with the Ingress/gateway when this moves to a cloud cluster (GKE), where a LoadBalancer + TLS make it meaningful.

### Fun resilience demos
```bash
kubectl -n novabot delete pod -l app=novabot-api --field-selector=status.phase=Running --force --grace-period=0
# watch K8s immediately recreate them; service stays up (that's why replicas: 2)
```

## Phase 6 — Real-Time Production Features (Redis cache, backpressure, A/B)

New in `serving/app/main.py` (v2.0.0) — all features degrade gracefully when Redis is absent, so unit tests and minimal setups still work:

- **Response cache (step 25):** deterministic requests (temperature 0, non-streaming) are cached in Redis keyed on (model, full message list, max_tokens), TTL 1h. Repeat questions return in ~ms without touching the GPU. Response includes `"cached": true/false`.
- **Redis rate limiting:** counters move to Redis (fixed 60s windows) so limits are correct across all replicas; falls back to the in-memory limiter if Redis is down.
- **Backpressure (step 24):** a semaphore caps concurrent engine calls per replica (`MAX_CONCURRENCY=8`); requests queue up to `QUEUE_TIMEOUT_S=10s`, then get a clean `503 Server busy` instead of piling onto the engine.
- **A/B testing (step 27):** set `MODEL_B_NAME` (+ optionally `ENGINE_B_BASE_URL`) and `AB_SPLIT_PERCENT` to route a deterministic % of users to variant B. Same `user_id` (or IP) always gets the same variant; responses and logs carry `"variant": "A"/"B"` for analysis. This is the mechanism for safely rolling out the Phase 8 retrained model.

### Deploy on the cluster
```bash
kubectl apply -f k8s/50-redis.yaml            # Redis (128MB, LRU eviction - cache semantics)
kubectl apply -f k8s/20-deployment.yaml       # picks up REDIS_URL + Phase 6 env
kubectl -n novabot rollout restart deploy/novabot-api
kubectl -n novabot get pods                   # redis + 2 api pods
```

### Verify
```bash
kubectl -n novabot port-forward svc/novabot-api 8080:80
# same question twice - second reply should show "cached": true and ~ms latency:
curl -s -X POST localhost:8080/v1/chat -H 'Content-Type: application/json' \
  -d '{"message": "What is the refund policy for monthly subscriptions?"}'
curl -s -X POST localhost:8080/v1/chat -H 'Content-Type: application/json' \
  -d '{"message": "What is the refund policy for monthly subscriptions?"}'
curl -s localhost:8080/readyz                 # now reports redis: up/down/disabled
```

## Phase 7 — Observability & Reliability (Prometheus + Grafana + alerts + feedback)

New in `serving/app/main.py` (v3.0.0):
- **`/metrics` endpoint (step 29):** Prometheus format. Metrics: `novabot_http_requests_total` (every response, by path/status — via middleware), `novabot_chat_requests_total` (by variant/cached/status), `novabot_chat_latency_seconds` histogram, `novabot_answer_chars` histogram (**drift signal**, step 31 — a shifting answer-length distribution flags model behavior change), `novabot_feedback_total`.
- **`POST /v1/feedback` (step 32):** `{request_id, rating: up|down, variant, comment}` → counted in metrics + logged as JSON. These logs are the raw material for Phase 8's retraining dataset.

New in `k8s/60-monitoring.yaml`:
- **Prometheus** — discovers api pods via Kubernetes service discovery (RBAC included), scrapes `/metrics` every 15s, evaluates **alert rules** (step 30): HighErrorRate (>5% 5xx), HighLatencyP95 (>10s uncached), ApiDown, NegativeFeedbackSpike (thumbs-down rate doubles = model quality alarm). View at `/alerts`. (Alertmanager routing to Slack/email = cloud phase.)
- **Grafana** — auto-provisioned with the Prometheus datasource and a **NovaBot Overview dashboard**: request rate by status, p95 latency (cached vs not), cache hit ratio, A/B traffic split, feedback up/down, answer-length drift, backpressure rejections.

### Deploy
```bash
# code changed -> rebuild API image inside minikube/kind, then:
kubectl apply -f k8s/60-monitoring.yaml
kubectl -n novabot rollout restart deploy/novabot-api
kubectl -n novabot get pods        # + prometheus, grafana pods
```
